# Check_check: контракт загрузки Excel для FE

## Endpoint

`POST /api/chef/check_check/upload_excel`

Использовать `multipart/form-data`.

Поля:

- `file`: обязательный файл отчета эквайринга `.xls` или `.xlsx`.
- `data`: обязательная JSON-строка или объект с теми же фильтрами, что и `get_orders`.

## Данные запроса

```json
{
  "date_start": "2026-05-01",
  "date_end": "2026-05-07",
  "point": {
    "id": 1,
    "base": "jaco_rolls_1",
    "name": "Тольятти, Ленинградская 47"
  },
  "kassa": [
    { "id": 1, "name": "Касса 1" },
    { "id": 2, "name": "Касса 2 (онлайн)" },
    { "id": 3, "name": "Касса 3" }
  ],
  "compare_with": "chef"
}
```

Примечания:

- `compare_with` необязателен. Допустимые значения: `chef`, `ofd`. По умолчанию `chef`.
- Excel эквайринга содержит только банковские операции, поэтому сравниваются только суммы банка.
- Касса `2` считается онлайн-кассой и игнорируется при сравнении Excel. Backend возвращает ее в `upload.ignored_kassas`.
- Excel может содержать месяц и несколько точек. Backend фильтрует строки по выбранному периоду и точке.
- Точка сначала ищется по `Номер терминала`, если он есть в DB, затем fallback по нормализованному адресу Excel против `points.addr/full_addr`.

## Ошибка

```json
{
  "st": false,
  "text": "Необходимо загрузить Excel файл"
}
```

Возможные значения `text`:

- `Необходимо указать все даты`
- `Необходимо выбрать точку`
- `Необходимо выбрать кассу`
- `Необходимо выбрать физическую кассу`
- `Необходимо загрузить Excel файл`
- `Поддерживаются только xls и xlsx`
- `В Excel не найдены обязательные колонки: Адрес ТСТ, Номер терминала, Дата операции, Тип операции, Сумма операции`
- `В Excel нет строк для выбранной точки и периода`
- `Не удалось определить кассу по терминалам Excel для выбранного периода`
- `Ошибка чтения Excel файла`

## Правила `Тип операции`

Колонка Excel `Тип операции` используется только для знака суммы эквайринга:

- значения с `Покупка` учитываются как положительная банковская сумма;
- значения с `возврат` или `отмена` учитываются как отрицательная банковская сумма;
- неизвестные типы операций пропускаются и возвращаются в `warnings.unknown_operation_types`.

Текущий пример Excel содержит только:

| Тип операции              | Строк | Знак парсера  |
| ------------------------- | ----: | ------------- |
| `Покупка`                 | 10380 | положительный |
| `Покупка СБП - стороннее` |    18 | положительный |

В `get_orders` нет эквивалентной колонки типа операции эквайринга. Текущая агрегация работает так:

- данные OFD/check используют `order_info_test.type_check`: `1` прибавляет сумму, `2` вычитает сумму.
- данные Chef берутся из успешных неудаленных заказов `orders`: `status_order = 6`, `is_delete = 0`.
- банковская сумма Chef: `orders.type_pay = 2`, сумма `summ_promo + summ_div`.
- наличная сумма Chef: `orders.type_pay = 1`, с исключениями доставок в offline-логике.

То есть Excel `Тип операции` концептуально соответствует знаку OFD `type_check`, а не Chef `type_pay`.

## Фильтрация кассы из Excel

В примере Excel нет фискальных значений ФН/ККТ. Проверено по апрельским значениям точки 1 из `jaco_main_rolls.kkt_info`:

- kassa `1`: KKT `0002280792002450`, FN `7380440902375610`
- kassa `3`: KKT `0004308225011499`, FN `7380440801863679`

Полезные колонки Excel содержат только банковские идентификаторы эквайринга: `Номер терминала`, `Номер мерчанта`, `MID головного терминала`, `TID головного терминала`. Они напрямую не связаны с Chef kassa в DB.

Поэтому backend выводит `Номер терминала -> kassa` заново на каждый upload-запрос: суммирует терминалы Excel за выбранный период и подбирает их к существующему Chef `summ_chef.kass_total` для выбранной точки, периода и касс. Пример для точки 1 / апреля 2026:

| Касса | Терминалы Excel                    | Банк Excel/Chef |
| ----: | ---------------------------------- | --------------: |
|   `1` | `10432448`, `26953272`, `29491623` |       `1869162` |
|   `3` | `21569793`, `26956677`, `29765140` |       `1990786` |

Когда FE выбирает только кассу `1` или только кассу `3`, строки другой выведенной группы терминалов пропускаются до расчета diff. В `warnings` возвращаются `kassa_inference_used`, `kassa_terminal_map`, `rows_skipped_by_kassa`.

Логика не привязана к этим конкретным terminal ID. При смене периода backend заново берет только строки Excel внутри `date_start/date_end`, заново считает суммы по `Номер терминала` и заново пытается сопоставить их с `summ_chef.kass_total`. Если для выбранной одной кассы невозможно надежно вывести группу терминалов, endpoint вернет ошибку `Не удалось определить кассу по терминалам Excel для выбранного периода`, чтобы не сравнивать всю точку как одну кассу.

### Flow фильтрации кассы

1. FE отправляет текущие фильтры `date_start`, `date_end`, `point`, `kassa` и Excel-файл в `/api/chef/check_check/upload_excel`.
2. Backend читает Excel и оставляет только строки выбранного периода. Если Excel содержит месяц, а пользователь выбрал неделю, строки вне недели попадают в счетчик `warnings.rows_outside_period`.
3. Backend определяет точку строки: сначала по `Номер терминала` через DB-маппинг, если он есть, затем по нормализованному `Адрес ТСТ` против `jaco_main_rolls.points.addr/full_addr`.
4. Backend оставляет только строки выбранной точки. Строки других точек попадают в `warnings.rows_other_points`.
5. Backend считает банковскую сумму Excel по каждому `Номер терминала` внутри выбранной точки и периода. Знак суммы берется из `Тип операции`.
6. Backend получает Chef totals через существующий `get_chef_summ(...)`. Для offline-касс `get_chef_summ` сам раскрывает UI `kassa` в `dop_kassa` через `jaco_main_rolls.kkt_info`.
7. Если выбраны кассы `1` и `3` вместе, Excel сравнивается как сумма выбранных physical-касс. Это покрывает сценарий общей сверки точки.
8. Если выбрана только одна physical-касса, backend пытается вывести, какие Excel-терминалы принадлежат этой кассе: ищет subset терминалов, сумма которого совпадает с `summ_chef.kass_total[*].summ_bank` выбранной кассы.
9. Если subset найден, строки остальных терминалов пропускаются до расчета diff. Количество таких строк возвращается в `warnings.rows_skipped_by_kassa`, а выведенная карта возвращается в `warnings.kassa_terminal_map`.
10. Если subset не найден для одиночной кассы, endpoint возвращает ошибку `Не удалось определить кассу по терминалам Excel для выбранного периода`. FE должен показать ее пользователю и предложить выбрать обе physical-кассы или другой период.
11. Касса `2` не участвует в Excel-сверке, потому что это online kassa. Если FE отправил ее вместе с другими кассами, backend игнорирует ее и возвращает `upload.ignored_kassas: [2]`.

## Excel на вкладке смен

`get_orders` уже возвращает смены в двух местах:

- `summ_ofd.smena`: агрегаты OFD по `date`, `kassa`, `smena`.
- `summ_chef.smena`: агрегаты Chef по `date`, `kassa`, `smena`.

Upload Excel теперь возвращает такие же ключи для отдельной вкладки:

- `excel.smena`: агрегаты Excel по `date`, `kassa`, `smena`.
- `diff.smena`: сравнение Excel против Chef/OFD по тем же `date`, `kassa`, `smena`.

Контракт строки `excel.smena[*]`:

```json
{
  "date": "2026-04-01",
  "kassa": 1,
  "smena": 12345,
  "summ_bank": 123456.78,
  "count_bank_checks": 120
}
```

Контракт строки `diff.smena[*]`:

```json
{
  "date": "2026-04-01",
  "kassa": 1,
  "smena": 12345,
  "excel_bank": 123456.78,
  "chef_bank": 123456.78,
  "ofd_bank": 123456.78,
  "diff_chef_bank": 0,
  "diff_ofd_bank": 0,
  "status": "ok",
  "color": "green"
}
```

Как определяется смена для Excel:

1. Excel не содержит номер смены.
2. Backend берет время операции из `Локальная дата операции`.
3. Backend уже знает кассу строки через flow фильтрации кассы по `Номер терминала`.
4. Backend читает смены из `jaco_main_rolls.cafe_smena_log_test` по выбранным `point_id`, `kassa`, `date_start/date_end`.
5. Excel-строка попадает в смену, если время операции находится между `date_time_open` и `date_time_close`.
6. Если смену определить нельзя, строка остается в дневных/общих итогах, но не попадает в `excel.smena`; счетчик таких строк возвращается в `warnings.rows_without_smena`.

Для FE: на вкладке смен можно использовать `diff.smena` как готовый источник строк с цветом. Если нужна исходная сумма Excel без diff, использовать `excel.smena`.

## Карта терминалов тестового Excel

Исходный Excel сейчас лежит в модуле:

- `Общий_отчёт_все_поля_6321390811_04052026_084630509.xlsx`

Период данных в Excel:

- `2026-04-01` - `2026-04-30`

Важно: эти значения Excel являются ID терминалов эквайринга, а не фискальными номерами ККТ. Они не найдены в `jaco_main_rolls.kkt_info.number`, поэтому этот файл матчится через fallback по адресу.

Таблицы DB, которые использует текущий matcher:

- попытка по терминалу: `jaco_main_rolls.kkt_info.number -> kkt_info.point_id -> jaco_main_rolls.points.id`
- fallback по адресу: нормализованный Excel `Адрес ТСТ` против `jaco_main_rolls.points.addr` и `jaco_main_rolls.points.full_addr`

Расчет `get_orders` использует другой KKT-маппинг:

- UI kassa IDs: `1`, `2`, `3`.
- online UI kassa `2` раскрывается из `jaco_main_rolls.kkt_info` по `point_id`, `base LIKE '%online%'` и выбранному периоду.
- offline-тоталы Chef используют `jaco_main_rolls.kkt_info.kassa`, чтобы найти `dop_kassa` для выбранной точки и периода.
- Excel `Номер терминала` не участвует в логике `get_orders`.

## Prompt для FE UX

Добавить загрузку Excel на существующий экран `check_check` после выбора фильтров.

Сценарий:

1. Пользователь выбирает точку, период и кассы как сейчас.
2. Пользователь загружает Excel эквайринга Сбербанка.
3. FE отправляет `multipart/form-data` на `/api/chef/check_check/upload_excel` с текущим фильтром `data` и `file`.
4. Показать summary из `diff.stats`: проверенные дни, расхождения, пропущенные строки, строки других точек, unmatched addresses.
5. Добавить колонку банка Excel рядом с текущими банковскими тоталами Chef/OFD.
6. Использовать `diff.days[*].color`: `green` для совпадения, `red` для расхождения.
7. Для каждого дня показать `excel_bank`, `chef_bank`, `ofd_bank`, `diff_chef_bank`, `diff_ofd_bank`.
8. Если `upload.ignored_kassas` содержит `2`, показать маленькое предупреждение, что online kassa игнорируется для Excel эквайринга.
9. Если `warnings.unmatched_addresses` не пустой, показать раскрываемый список предупреждений.

Не сохранять загруженный Excel на FE как сущность. Держать распарсенный ответ в состоянии компонента и требовать повторную загрузку после изменения фильтров.
