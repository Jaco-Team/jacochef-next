# BUSINESS PROCESSES

Статус: исследование текущей бизнес-логики складского контура и целевые правила для нового модуля `sklad`.

Основание:

- код старых модулей `sklad_items_module_new`, `recept_module_new_2`, `site_items_new`
- код формирования PDF БЖУ/составов
- текущая prod-БД

---

## 1. Термины

### `Item`

`Item`:

- таблица `jaco_main_rolls.items`
- это ингредиент / сырье / закупаемая складская позиция
- это не продаваемый продукт
- это upstream-уровень для остальной цепочки и одновременно часть текущего scope нового `sklad`

Связанные данные:

- единица измерения
- категория ингредиента
- места хранения
- учетные признаки
- прямые аллергены
- возможные аллергены

Прямые аллергенные связи ингредиента лежат в:

- `jaco_main_rolls.items_allergens_new`
- `jaco_main_rolls.items_allergens_other_new`

### `SemiFinished`

`SemiFinished`:

- таблица `jaco_main_rolls.polufabricat`
- это заготовка / полуфабрикат

Связанные данные:

- состав из `Item`
- собственные поля `structure`
- собственные поля `allergens`
- собственные поля `allergens_diff`
- отдельные аллергенные связи в link-таблицах

Состав заготовки хранится в:

- `jaco_main_rolls.polufabricat_items_new`

Аллергены заготовки участвуют в downstream-витрине через:

- `jaco_main_rolls.polufabricat_allergens`
- `jaco_main_rolls.polufabricat_allergens_other`

### `Recipe`

`Recipe`:

- таблица `jaco_main_rolls.recipes`
- это рецепт, который собирается из заготовок

Состав рецепта хранится в:

- `jaco_main_rolls.recipe_items_new`

Фактическое правило по БД:

- все `recipe_items_new.item_id` ссылаются на `jaco_main_rolls.polufabricat`

Значит рецепт в текущей модели:

- не собирается напрямую из `Item`
- опирается на `SemiFinished`

### `SiteItem`

`SiteItem`:

- таблица `jaco_site_rolls.items`
- это товар сайта / продаваемый продукт

Связанные composition-источники:

- `jaco_site_rolls.items_pf_new` -> прямые заготовки
- `jaco_site_rolls.items_rec_new` -> прямые рецепты
- `jaco_site_rolls.items_all_pf_new` -> производный агрегат для списаний

---

## 2. Фактическая бизнес-цепочка

Текущая доменная цепочка в коде и БД выглядит так:

`Item -> SemiFinished -> Recipe -> SiteItem`

При этом у `SiteItem` есть и прямой путь:

`Item -> SemiFinished -> SiteItem`

То есть `SiteItem` может включать:

- заготовки напрямую
- рецепты, которые внутри уже состоят из заготовок

Это подтверждается текущими таблицами:

- `polufabricat_items_new` целиком связывает `pf -> item`
- `recipe_items_new` целиком связывает `recipe -> pf`
- `items_pf_new` связывает `site_item -> pf`
- `items_rec_new` связывает `site_item -> recipe`

---

## 3. Как сейчас собирается состав продукта

В старом коде `SiteItem` не хранит итоговый состав как первичную бизнес-сущность.

Сейчас процесс такой:

1. Пользователь сохраняет stage-состав продукта.
2. Сохраняются stage-строки:

- `items_pf_stages_new`
- `items_rec_stages_new`

3. Из stage-строк пересобираются плоские связи:

- `items_pf_new`
- `items_rec_new`

4. После этого пересобирается производный агрегат:

- `items_all_pf_new`

Назначение `items_all_pf_new`:

- агрегированная заготовочная структура для списаний и downstream-операций

Практический вывод для нового модуля:

- изменение состава продукта должно сопровождаться пересборкой производных связей
- нельзя ограничиться апдейтом только одной таблицы карточки товара

---

## 4. Как сейчас собираются итоговые аллергены продукта

В текущем контуре итоговые аллергены для продуктовой витрины собираются не из `SiteItem`, а из заготовок.

Это видно по сервису формирования PDF БЖУ/составов:

- `SiteItem` получает composition rows из `items_pf` и `items_rec`
- для `items_rec` рецепт разворачивается через `recipe_items` в `polufabricat`
- затем для всех участвующих `pf_id` подтягиваются:
  - базовые аллергены из `polufabricat_allergens`
  - возможные аллергены из `polufabricat_allergens_other`

Итог:

- текущая публичная витрина состава/аллергенов продукта строится на уровне `SemiFinished`
- `Recipe` служит промежуточной структурой
- `Item` в этом downstream-процессе напрямую не участвует

---

## 5. Что это значит для задачи про перенос аллергенов

Фраза задачи:

- “аллергены, состав и возможные аллергены переносим из заготовок в товар”

В доменной интерпретации не должна означать:

- просто скопировать текстовые поля `polufabricat.allergens` в `SiteItem`
- просто завести ручное поле аллергенов на `SiteItem` и забыть про источник

Корректная интерпретация:

- новый модуль должен показать и обслуживать итоговые аллергенные данные товара как производный результат его состава
- первичный источник аллергенной семантики начинается ниже, на уровне ингредиентов и заготовок
- при наличии аллергенной привязки у `Item` должна существовать явная бизнес-логика, как она попадает в итоговую аллергенную картину `SiteItem`

На текущем legacy-контуре виден разрыв:

- у `Item` есть прямые аллергенные связи
- downstream-витрина продукта читает аллергены только с `SemiFinished`
- автоматической трассируемой агрегации `Item allergens -> PF allergens -> SiteItem allergens` в найденном коде не видно

Следствие:

- это не просто UI-перенос полей
- это отдельное бизнес-правило нового модуля, которое надо зафиксировать явно

Текущее решение в новом модуле:

- производный состав и производные аллергены вынесены в отдельный `SkladDerivedStateService`
- `SiteItem`, `Recipe` и `SemiFinished` читают derived state через этот слой, а не держат параллельные SQL-реализации в своих read-services
- это сохраняет текущую БД и текущую бизнес-цепочку, но делает правило явным и расширяемым

---

## 6. Целевое правило для нового модуля

Рекомендуемое целевое правило:

1. `Item` хранит первичные аллергенные связи:

- `allergens`
- `possible_allergens`

2. `SemiFinished` показывает:

- собственный состав
- итоговые аллергены, вычислимые по составу ингредиентов
- при необходимости ручные overrides, если бизнес их реально требует

3. `Recipe` показывает:

- состав из заготовок
- итоговые аллергены как производную от связанных заготовок

4. `SiteItem` показывает:

- состав по прямым `pf` и `recipe`
- итоговые аллергены как агрегат по всему составу
- возможные аллергены как агрегат по всему составу

5. Для API должны быть разделены:

- source fields
- derived fields
- manual override fields, если они останутся

---

## 7. Что новый модуль обязан сделать явно

Новый `sklad` должен описать в коде и контракте:

- откуда берется итоговый состав `SiteItem`
- откуда берутся итоговые аллергены `SiteItem`
- когда эти данные считаются
- где они пересчитываются при изменении состава
- какие поля являются первичными
- какие поля являются производными
- допускаются ли ручные корректировки итоговых аллергенов

Без этого перенос “из заготовок в товар” останется не бизнес-процессом, а скрытой копией данных.

---

## 8. Конвертация recipe <-> semi_finished

Целевая реализация в новом `sklad` должна считать конвертацию не rename-операцией, а business transformation:

- создается новая сущность целевого типа
- переносится карточка, состав, места хранения, роли приготовления и история стартового состояния новой сущности
- исходная сущность удаляется только если ее реально можно удалить без разрушения связанных контуров

Практическое правило для текущей схемы БД:

- `semi_finished -> recipe` является lossless, потому что `item`-состав полуфабриката можно напрямую перенести в `recipe_items_new` как `type = item`
- `recipe -> semi_finished` является lossless только для item-based рецептов

Причина ограничения:

- `polufabricat_items_new` хранит только связь `pf -> item`
- если рецепт содержит `pf` или `rec` компоненты, прямое копирование в полуфабрикат разрушает семантику состава

Следствие для API:

- конвертация `recipe -> semi_finished` должна явно отклоняться для рецептов, чей состав не является item-based
