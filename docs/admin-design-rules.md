# Admin Design Rules

Этот документ фиксирует текущий дизайн-язык обновлённых страниц и нужен как краткий набор правил для следующих задач.

## Основная идея

В проекте сейчас есть 3 устойчивых UI-режима:

1. `content/admin`
   Использовать для CRUD-страниц, справочников, контента и настроек.
   Основные референсы: `site_items_new`, `job_descriptions`.

2. `dashboard`
   Использовать для стартовых экранов, overview и навигационных подборок.
   Основной референс: `index`.

3. `journal/operational`
   Использовать для журналов, операций, списаний и прикладных форм.
   Основной референс: `write_off_journal_2`.

Перед началом новой задачи сначала определить, к какому режиму относится экран, и не смешивать несколько режимов внутри одного интерфейса без явной причины.

## Базовые токены

Для `content/admin` по умолчанию использовать:

- `brandRed: #DD1A32`
- `blockBackground: #F3F3F3`
- `blockBorder: #E5E5E5`
- `textPrimary: #3C3B3B`
- `textSecondary: #5E5E5E`
- `muted: #A6A6A6`
- `placeholder: #BABABA`

Дополнительные правила:

- Красный использовать как основной акцент для действий, активных состояний и secondary outline-кнопок.
- Синий не использовать как глобальный brand-accent на admin-страницах. Он допустим в `dashboard`.
- Зелёный использовать только в `journal/operational`-сценариях как цвет прикладного действия.

## Режимы страниц

### `content/admin`

Использовать для страниц наподобие `site_items_new` и `job_descriptions`.

Характерные признаки:

- крупный page title;
- справа или под ним стек действий;
- основной контент собран в крупные серые секции;
- внутри секций белые карточки, таблицы, списки и формы;
- почти нет теней, визуальная иерархия строится на фоне, границе и отступах;
- mobile-версия не сжимает desktop-таблицы, а перестраивается в карточки или sheet.

### `dashboard`

Использовать для страниц наподобие `index`.

Характерные признаки:

- сетка интерактивных карточек;
- больше hover- и drag-feedback;
- допустимы локальные синие акценты;
- отдельный боковой или нижний блок обновлений;
- карточки могут быть более живыми и “навигационными”, чем на content-страницах.

### `journal/operational`

Использовать для страниц наподобие `write_off_journal_2`.

Характерные признаки:

- прикладные фильтры и рабочие формы в стиле `journal`;
- зелёный как основной цвет действия;
- большие рабочие модалки и полноэкранные режимы;
- таблицы и сводки ориентированы на операцию, а не на маркетинговое представление данных.

Если делается новый журнал или операционный экран, ориентироваться именно на этот режим, а не на `content/admin`.

## Layout

- На desktop использовать крупный page-shell: заголовок, блок действий, затем одна или несколько секций.
- На `content/admin` секции чаще всего имеют серый фон и внутренние белые поверхности.
- На mobile секции допускается выводить во всю ширину экрана через отрицательные внешние отступы и без внешнего радиуса.
- Не плодить мелкие вложенные карточки без необходимости.
- Не делать лишние внутренние scroll-area, если экран может нормально жить в page scroll.

## Поверхности и радиусы

- Базовый радиус для карточек, таблиц и блоков: `12px`.
- Для полей допустим диапазон `12px` - `18px` в зависимости от паттерна.
- Для sheet на mobile использовать более крупный верхний радиус, около `24px` - `25px`.
- Pill-форма (`999px`) подходит для счётчиков, компактных фильтров и статусных капсул.
- Обычные admin/content-блоки не должны опираться на тяжёлые тени.

## Типографика

- Основной заголовок страницы: спокойный, крупный, без избыточной жирности.
- Заголовки секций: около `20px / 24px`.
- Текст действий и строк интерфейса: около `16px / 20px`.
- Вспомогательные подписи и label-value блоки: `11px` - `14px`.
- Не использовать агрессивный uppercase для обычного UI.
- Uppercase допустим только для компактных label внутри карточек и таблиц.

## Кнопки

Общие правила:

- `textTransform: "none"` по умолчанию.
- Основная высота кнопок: `40px` - `44px`.
- Радиус: `12px` или MUI `1.5` - `2`.
- Primary-кнопка: solid accent без box-shadow.
- Secondary-кнопка: белая `outlined`.
- destructive action должен быть визуально отделён от save action.

Для `content/admin`:

- primary: красная;
- secondary: outline красная или нейтральная белая с серой рамкой.

Для `journal/operational`:

- primary: зелёная;
- secondary/destructive: белая кнопка с цветным текстом и рамкой.

## Формы

- Сначала переиспользовать `MyTextInput`, `MyAutocomplite`, `MyDatePickerNew`, `TextEditor`.
- Не создавать одноразовые input-обёртки без необходимости.
- Для journal-экранов использовать `customRI="journal"`.
- Для новых content/admin-экранов использовать unified autocomplete, если поле открывает dropdown.
- Placeholder должен оставаться светло-серым, введённое значение читаемым и достаточно контрастным.
- Disabled-состояния должны быть мягкими, но не блеклыми до нечитаемости.

## Таблицы

- Таблицу использовать только там, где данные реально табличные.
- Header таблицы должен иметь спокойный серый фон.
- Основное тело таблицы белое, с тонкими разделителями.
- Колонки должны иметь управляемые ширины на desktop.
- Hover должен быть лёгким, без тяжёлого контраста.
- Сортировка должна быть заметной, но не кричащей.

На mobile:

- Не сжимать desktop-таблицу в узкий экран.
- Переводить строки в карточки с label/value-подачей.
- В карточке выделять заголовок сущности, статусы и блок действий.

## Списки и карточки

- Для item-list на content/admin-страницах использовать спокойные белые карточки внутри серого секционного контейнера.
- Для grouped-данных использовать `Accordion`, если это улучшает читаемость и не ломает flow.
- Для dashboard-карточек допустимы drag handle, hover и локальные теги/чипы.
- Для inline-actions в строках и карточках предпочитать иконки в небольших белых bordered-контейнерах.

## Модалки и sheet

Desktop:

- использовать `Dialog`;
- мягкий backdrop `rgba(0,0,0,0.3)`;
- paper без тяжёлой тени;
- аккуратный серый title bar;
- справа круглая кнопка закрытия;
- кнопки действий внизу, без визуального шума.

Mobile:

- если это редактирование сущности или полноценная форма, предпочитать `SwipeableDrawer` как bottom sheet;
- у sheet должен быть крупный верхний радиус;
- нужен drag handle;
- шапка sheet должна быть белой, а body может оставаться на сером фоне секции.

## История и подтверждения

- Не использовать browser `confirm`.
- Все destructive-действия проводить через styled confirmation dialog.
- Историю изменений показывать через `HistoryLog`.
- Историю располагать рядом с тем блоком, к которому она относится.

## Empty, loading, feedback

- Loading страницы: `Backdrop + CircularProgress`.
- Loading локального участка: компактный `CircularProgress` внутри блока.
- Empty-state: короткий спокойный текст внутри белого или секционного блока.
- Ошибки и статусы оставлять в текущем проектном паттерне через `MyAlert`.

## Что не делать

- Не смешивать в одном экране красный content-style, синий dashboard-style и зелёный journal-style без явной причины.
- Не пытаться сделать один универсальный layout для всех страниц.
- Не заменять mobile-карточки ужатой desktop-таблицей.
- Не строить иерархию только тенями.
- Не делать кнопки в uppercase по умолчанию.
- Не тащить старые native-confirm или случайные browser-like паттерны в новый UI.

## Быстрый выбор референса

Если задача про:

- товары, тэги, контент, справочники, инструкции, новости:
  брать `site_items_new` и `job_descriptions`;
- избранное, стартовую страницу, модульный обзор:
  брать `index`;
- журнал, списание, операционную форму, акт:
  брать `write_off_journal_2`.
