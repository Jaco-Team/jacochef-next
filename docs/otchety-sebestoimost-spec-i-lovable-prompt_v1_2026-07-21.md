# Отчёты по себестоимости товаров и готовых блюд

Версия: 1.0  
Дата: 21.07.2026  
Назначение: аналитическая записка, спецификация источников и расчётов, UX-требования и готовый промт для генерации интерфейса в Lovable.

## 1. Результат анализа

Нужен новый модуль «Отчёты» с двумя самостоятельными отчётами:

1. «Продажи товаров» — для покупных товаров, которые кафе перепродаёт без производства: напитки, готовые соусы и другие позиции.
2. «Производство и продажи готовых блюд» — для блюд и соусов собственного производства, себестоимость которых рассчитывается через рецептуры, вложенные полуфабрикаты, потери и закупочные цены исходных продуктов.

Общий принцип формирования себестоимости:

```text
Покупной товар
  └─ документы поступления товаров
       └─ закупочная себестоимость товара

Готовое блюдо
  └─ документ выпуска продукции
       └─ действующая на дату выпуска рецептура
            └─ рекурсивное разложение полуфабрикатов до исходных продуктов
                 └─ документы поступления продуктов
                      └─ закупочная себестоимость исходных продуктов
```

Себестоимость должна рассчитываться отдельно по кафе: закупочные цены, документы, рецептуры и выпуск могут отличаться. Объединение нескольких кафе допустимо только на уровне итогового отчёта после расчёта каждой точки.

## 2. Изученные материалы

### 2.1. Документ «отчеты.docx»

Документ описывает две таблицы:

- отчёт о розничных продажах товаров;
- отчёт производства и розничных продаж готовых блюд.

Параметры поиска предлагается взять из существующего модуля «Отчёт по выручке» и дополнить выбором позиций.

В исходном документе номера колонок и ссылки на них местами не совпадают. Поэтому backend и интерфейс должны опираться не на номера колонок, а на стабильные имена показателей.

### 2.2. Таблица «рецепты_с_выпуском_продукции.xlsx»

В книге четыре листа:

1. «процесс приготовления» — дерево рецептуры, холодные/горячие потери и разворачивание полуфабрикатов до исходных продуктов;
2. «Закупки» — документы поступления, количество, сумма и цена единицы;
3. «Средние цены» — взвешенная закупочная цена за 14 дней до начала отчётного периода;
4. «Выпуск продукции» — количество выпущенных блюд, расход исходных продуктов, стоимость каждого компонента, общая и удельная себестоимость.

Книга подтверждает следующие правила:

- средняя цена является взвешенной по количеству, а не средним арифметическим цен поставок;
- цена ищется за 14 календарных дней перед началом отчётного периода;
- рецептуры раскрываются рекурсивно до покупных продуктов;
- холодные и горячие потери учитываются на каждом уровне рецептуры;
- себестоимость выпуска равна сумме стоимости израсходованных исходных продуктов;
- себестоимость единицы равна себестоимости выпуска, делённой на количество выпущенной продукции.

## 3. Термины и границы отчёта

| Термин                | Значение                                                                          |
| --------------------- | --------------------------------------------------------------------------------- |
| Товар для перепродажи | Покупная позиция, продаваемая без преобразования по рецептуре                     |
| Исходный продукт      | Покупная складская позиция, используемая в рецептурах                             |
| Полуфабрикат          | Результат промежуточной рецептуры, который входит в другую рецептуру              |
| Готовое блюдо         | Конечный результат рецептуры, доступный для продажи                               |
| Выпуск                | Фактически зарегистрированное изготовление готовой продукции                      |
| Расчётный выпуск      | Временная оценка `продажи + подтверждённые списания`, если документов выпуска нет |
| Себестоимость         | Стоимость фактически закупленных компонентов, отнесённая на товар или блюдо       |
| Торговая наценка      | `(цена продажи − себестоимость) / себестоимость × 100%`                           |
| Валовая маржа         | `(выручка − себестоимость продаж) / выручка × 100%`                               |

Наценку и маржу нельзя смешивать: у них разные знаменатели и разный экономический смысл.

## 4. Источники данных

### 4.1. Фильтры и права доступа

| Данные                     | Источник                                              | Правило                                                                            |
| -------------------------- | ----------------------------------------------------- | ---------------------------------------------------------------------------------- |
| Доступные кафе             | текущая система прав пользователя                     | Backend самостоятельно ограничивает точки; присланным с клиента ID доверять нельзя |
| Город, адрес, организация  | существующий справочник кафе                          | Использовать текущую группировку `CityCafeAutocomplete2`                           |
| Даты                       | пользовательский фильтр                               | Проверять `date_start <= date_end` и допустимый лимит периода                      |
| Категории                  | существующий справочник категорий продаж/номенклатуры | Возвращать только категории, содержащие доступные позиции                          |
| Позиции                    | товары для перепродажи либо конечные блюда            | Зависит от выбранного вида отчёта                                                  |
| Доступ к модулю и экспорту | отдельные права `access` и `export_items`             | Повторить паттерн текущего `report_revenue`                                        |

Текущий frontend вызывает Laravel через:

```js
api_laravel("report_revenue", "get_all");
api_laravel("report_revenue", "get_report", payload);
```

Существующий запрос отчёта по выручке содержит:

```json
{
  "date_start": "2026-07-01",
  "date_end": "2026-07-31",
  "points": [],
  "cats": []
}
```

Для нового модуля можно сохранить этот транспортный паттерн, но создать отдельный API-модуль, например `cost_reports`, чтобы не менять расчёты legacy-отчёта по выручке.

### 4.2. Продажи и выручка

Первичный прикладной источник — существующий backend модуля `report_revenue`:

- `get_all` — доступные точки, категории, название модуля и права;
- `get_report` — продажи и выручка за период;
- в строках используются как минимум `item.id`, `item.name`, `data[].count`, `data[].price`, `total.count`, `total.price`;
- отдельно присутствуют ценовые сегменты и показатели продаж со скидкой.

Для нового backend лучше использовать тот же серверный слой расчёта продаж, а не читать готовый JSON другого API-метода и не дублировать бизнес-логику. Конкретные таблицы заказов и статусы необходимо взять из реализации `report_revenue` в legacy/backend. Их нельзя назначать по догадке: должны сохраниться те же исключения отменённых заказов, возвратов, скидок, промокодов и служебных позиций.

Обязательные результирующие поля по позиции и кафе:

- `sold_quantity` — количество проданных единиц;
- `revenue` — фактическая выручка после применённых скидок по действующим правилам отчёта;
- `average_sale_price = revenue / sold_quantity`.

Позиции с нулевыми продажами следует возвращать, если включён флаг `include_zero_sales`; это прямо требуется для отчёта готовых блюд.

### 4.3. Закупки и цена

Подтверждённые ранее таблицы предметной области:

- `billing` — заголовок документа поступления;
- `billing_items` — строки документа;
- `items` — сопоставление закупочной позиции с товаром/продуктом, единица измерения и коэффициенты;
- справочники единиц измерения — для приведения в базовую единицу.

В расчёт допускаются только действующие/проведённые поступления. Необходимо сохранить существующую семантику:

- возврат поставщику уменьшает количество и стоимость;
- корректировка количества изменяет знаменатель;
- корректировка суммы изменяет числитель;
- отменённый или неподтверждённый документ не участвует;
- количество приводится к базовой единице товара;
- цена берётся в единой договорённой трактовке НДС.

Рекомендуется использовать фактическую цену с НДС, так как в текущей аналитике ревизий применяется именно она. Это решение нужно подтвердить с бухгалтерией до разработки.

### 4.4. Рецептуры и потери

Подтверждённые справочники предметной области:

- `polufabricat` — заготовки/полуфабрикаты;
- `category_pf` — категории;
- `ed_izmer` — единицы измерения;
- `items` — покупные товары и продукты.

Точные таблицы заголовков рецептур, компонентов, версий и процентов холодной/горячей обработки необходимо подтвердить в legacy-схеме перед реализацией. В API-слое нужны следующие логические поля независимо от фактических названий колонок:

```text
recipe_id
recipe_version_id
valid_from / valid_to
result_item_id
result_quantity
component_type: item | semifinished
component_id
gross_quantity
cold_loss_percent
hot_loss_percent
unit_id
```

Версия рецептуры выбирается по дате выпуска. Текущую рецептуру нельзя применять задним числом, если за отчётный период она изменялась.

### 4.5. Выпуск, списания и продажи готовых блюд

Приоритет источников:

1. документ фактического выпуска продукции — основной и рекомендуемый источник;
2. подтверждённый `write_off_journal` — отдельный источник списаний;
3. продажи — существующий расчёт `report_revenue`;
4. `продажи + списания` — только временная оценка выпуска при отсутствии документов выпуска.

`order_promo` в списания не включается: по уточнённому правилу списания ведутся только в `write_off_journal`.

Нельзя называть `продажи + списания` фактическим выпуском. Такая формула не учитывает остаток готовой продукции на начало и конец периода, внутренние перемещения, возвраты в производство и незарегистрированные потери. Если временный режим всё же используется, API обязан вернуть:

```json
{
  "production_source": "estimated_sales_plus_writeoffs",
  "is_estimated": true,
  "warning": "Выпуск рассчитан по продажам и списаниям, а не по документам производства"
}
```

## 5. Расчёт закупочной цены

### 5.1. Период цены из макета

В таблице цена берётся за 14 полных календарных дней перед началом отчёта:

```text
price_date_from = date_start − 14 дней
price_date_to   = date_start
интервал        = [price_date_from, price_date_to)
```

Например, для `date_start = 10.07.2026` используются поступления с 26.06.2026 по 09.07.2026 включительно.

### 5.2. Взвешенная цена

```text
weighted_purchase_price = net_purchase_amount / net_purchase_quantity
```

Где количество и сумма уже учитывают возвраты, корректировки и перевод в базовую единицу.

Пример из таблицы по рису:

```text
50 кг × 20 ₽    = 1 000 ₽
40 кг × 12,50 ₽ =   500 ₽

Средняя цена = 1 500 ₽ / 90 кг = 16,6667 ₽/кг
```

Среднее арифметическое `(20 + 12,5) / 2 = 16,25` использовать нельзя, потому что объёмы поставок различаются.

### 5.3. Если в 14-дневном окне нет цены

Рекомендуемое правило:

1. искать последнее проведённое поступление до `date_start` без ограничения по давности;
2. вернуть фактическую дату и источник цены;
3. показать предупреждение, если цена старше 14 дней;
4. если цены нет вообще — вернуть `unit_cost: null`, а не ноль.

Это сохраняет возможность построить отчёт и одновременно показывает качество данных.

Для каждой цены API должен вернуть:

```json
{
  "unit_cost": 16.666667,
  "cost_currency": "RUB",
  "price_method": "weighted_average_14_days",
  "price_date_from": "2026-06-26",
  "price_date_to": "2026-07-10",
  "last_receipt_date": "2026-06-28",
  "receipt_count": 2,
  "net_quantity": 90,
  "net_amount": 1500,
  "is_fallback_price": false
}
```

## 6. Расчёт рецептуры

### 6.1. Базовые формулы потерь

Для каждого компонента:

```text
net_after_cold = gross × (1 − cold_loss_percent / 100)
output_after_hot = net_after_cold × (1 − hot_loss_percent / 100)
```

Проценты могут быть отрицательными. Например, `−100%` означает увеличение выхода в два раза:

```text
3 кг × (1 − (−100 / 100)) = 6 кг
```

Backend не должен ограничивать потери диапазоном `0…100`, пока бизнес не введёт отдельную модель влагонабора/привеса.

### 6.2. Масштабирование рецептуры на требуемый выпуск

```text
scale = required_output / recipe_output
required_component_gross = recipe_component_gross × scale
```

Если компонент является полуфабрикатом, его количество рекурсивно раскрывается по его собственной рецептуре до покупных продуктов.

### 6.3. Пример из таблицы

Рецептура «Рис роллы»:

- 6,000 кг варёного риса;
- 0,800 кг мицукана;
- по 5% горячих потерь;
- выход 6,460 кг;
- на один ролл требуется 0,150 кг готового риса.

Первый уровень:

```text
Варёный рис = 0,150 / 6,460 × 6,000 = 0,139319 кг
Мицукан     = 0,150 / 6,460 × 0,800 = 0,018576 кг
```

После рекурсивного раскрытия до покупных продуктов на один ролл получается приблизительно:

| Продукт | Количество |
| ------- | ---------: |
| Рис     |   0,070 кг |
| Вода    |   0,074 кг |
| Уксус   |   0,009 кг |
| Сахар   |   0,009 кг |
| Соль    |   0,001 кг |

Себестоимость продукта:

```text
component_cost = raw_quantity × weighted_purchase_price
```

Себестоимость блюда:

```text
dish_unit_cost = Σ component_cost
```

По примеру таблицы получается около `3,39 ₽` на ролл.

### 6.4. Защита рекурсивного расчёта

Backend обязан:

- обнаруживать циклы рецептур;
- ограничивать глубину вложенности техническим лимитом;
- не смешивать единицы измерения без явного коэффициента;
- сохранять путь разложения для проверки;
- возвращать ошибку по конкретной позиции, а не обнулять весь отчёт;
- выбирать версию каждого вложенного полуфабриката на ту же дату выпуска.

## 7. Формулы отчётов

### 7.1. Товары для перепродажи

| Поле API             | Колонка интерфейса    | Формула/источник                           |
| -------------------- | --------------------- | ------------------------------------------ |
| `item_name`          | Товар                 | справочник товаров                         |
| `sold_quantity`      | Продано               | отчёт по выручке                           |
| `unit_cost`          | Себестоимость единицы | взвешенная закупочная цена                 |
| `cost_of_goods_sold` | Себестоимость продаж  | `sold_quantity × unit_cost`                |
| `average_sale_price` | Средняя цена продажи  | `revenue / sold_quantity`                  |
| `revenue`            | Выручка               | отчёт по выручке                           |
| `gross_profit`       | Валовая прибыль       | `revenue − cost_of_goods_sold`             |
| `markup_percent`     | Наценка               | `gross_profit / cost_of_goods_sold × 100%` |
| `margin_percent`     | Маржа                 | `gross_profit / revenue × 100%`            |

При нулевых продажах `average_sale_price`, проценты наценки и маржи возвращаются как `null`.

### 7.2. Готовые блюда

| Поле API               | Колонка интерфейса     | Формула/источник                                |
| ---------------------- | ---------------------- | ----------------------------------------------- |
| `item_name`            | Блюдо                  | справочник конечных блюд                        |
| `produced_quantity`    | Выпущено               | документ выпуска; временно — продажи + списания |
| `unit_cost`            | Себестоимость единицы  | рекурсивный расчёт рецептуры                    |
| `production_cost`      | Себестоимость выпуска  | `produced_quantity × unit_cost`                 |
| `sold_quantity`        | Продано                | отчёт по выручке                                |
| `cost_of_goods_sold`   | Себестоимость продаж   | `sold_quantity × unit_cost`                     |
| `written_off_quantity` | Списано                | подтверждённый `write_off_journal`              |
| `write_off_cost`       | Себестоимость списаний | `written_off_quantity × unit_cost`              |
| `average_sale_price`   | Средняя цена продажи   | `revenue / sold_quantity`                       |
| `revenue`              | Выручка                | отчёт по выручке                                |
| `gross_profit`         | Валовая прибыль        | `revenue − cost_of_goods_sold`                  |
| `markup_percent`       | Наценка                | `gross_profit / cost_of_goods_sold × 100%`      |
| `margin_percent`       | Маржа                  | `gross_profit / revenue × 100%`                 |

### 7.3. Итоги

Проценты строк нельзя складывать или усреднять арифметически.

```text
total_gross_profit = total_revenue − total_cost_of_goods_sold
total_markup_percent = total_gross_profit / total_cost_of_goods_sold × 100%
total_margin_percent = total_gross_profit / total_revenue × 100%
```

Если часть строк не имеет себестоимости, нельзя подставлять ноль. Отчёт должен отдельно вернуть:

```text
known_cost_total
rows_with_cost
rows_without_cost
cost_coverage_percent
is_total_complete
```

Неполный денежный итог маркируется как «по позициям с известной себестоимостью».

## 8. Рекомендуемый backend-контракт

Предлагаемый модуль: `cost_reports`.

### 8.1. `get_all`

Возвращает:

- название модуля;
- права;
- доступные кафе;
- виды отчёта;
- категории и позиции для каждого вида;
- допустимые единицы измерения;
- доступность фактических документов выпуска.

### 8.2. `get_report`

Запрос:

```json
{
  "report_type": "resale_goods",
  "date_start": "2026-07-01",
  "date_end": "2026-07-31",
  "point_ids": [12, 18],
  "category_ids": [4, 9],
  "item_ids": [101, 205],
  "include_zero_sales": true
}
```

Для готовых блюд `report_type = "prepared_dishes"`.

Ответ верхнего уровня:

```json
{
  "st": true,
  "meta": {
    "report_type": "prepared_dishes",
    "date_start": "2026-07-01",
    "date_end": "2026-07-31",
    "generated_at": "2026-07-21T12:00:00+04:00",
    "cost_method": "weighted_average_14_days",
    "production_source": "production_documents",
    "is_estimated": false
  },
  "summary": {},
  "rows": [],
  "warnings": []
}
```

Строка отчёта должна содержать стабильные числовые поля, а не заранее отформатированные строки:

```json
{
  "point_id": 12,
  "point_name": "Ленинградская 47",
  "city_name": "Тольятти",
  "item_id": 501,
  "item_name": "Ролл Филадельфия",
  "unit": "шт",
  "produced_quantity": 120,
  "sold_quantity": 110,
  "written_off_quantity": 5,
  "unit_cost": 83.4256,
  "production_cost": 10011.072,
  "cost_of_goods_sold": 9176.816,
  "write_off_cost": 417.128,
  "average_sale_price": 349,
  "revenue": 38390,
  "gross_profit": 29213.184,
  "markup_percent": 318.33,
  "margin_percent": 76.1,
  "cost_status": "complete"
}
```

### 8.3. `get_cost_breakdown`

Drill-down одной строки. Возвращает:

- исходные документы продаж;
- источник выпуска;
- версию и дату рецептуры;
- дерево полуфабрикатов;
- брутто, холодные/горячие потери и выход каждого узла;
- конечные количества покупных продуктов;
- закупочную цену и документы, из которых она получена;
- подстановку значений в формулу.

### 8.4. `export_file_xls`

Экспорт должен повторять активные фильтры и возвращать:

- лист «Итоги»;
- лист «По кафе»;
- лист «Расшифровка себестоимости»;
- лист «Предупреждения».

На каждом листе указываются период, метод цены, дата формирования и признак расчётного выпуска.

## 9. Архитектура расчёта

Рекомендуемая последовательность backend:

1. Проверить право доступа и сервером разрешить список кафе.
2. Нормализовать фильтры и даты.
3. Получить продажи и выручку через общий серверный слой `report_revenue`.
4. Разделить покупные товары и готовые блюда по справочнику, а не по названию.
5. Получить проведённые поступления по каждому кафе и продукту.
6. Рассчитать взвешенные закупочные цены и сохранить диагностические данные.
7. Для готовых блюд выбрать версии рецептур по датам выпуска.
8. Рекурсивно раскрыть рецептуры с потерями и единицами измерения.
9. Получить фактический либо расчётный выпуск и подтверждённые списания.
10. Рассчитать строки отдельно по кафе.
11. Сформировать агрегаты из строк, не усредняя проценты.
12. Вернуть предупреждения и полноту данных.

Для повторных расчётов желательно кэшировать раскрытую рецептуру по ключу:

```text
point_id + recipe_version_id + output_unit + calculation_date
```

Закупочную цену кэшировать отдельно по:

```text
point_id + item_id + price_date_from + price_date_to
```

## 10. Контроль качества данных

Предупреждения должны быть структурированными:

| Код                       | Ситуация                         | Поведение                                              |
| ------------------------- | -------------------------------- | ------------------------------------------------------ |
| `MISSING_PURCHASE_PRICE`  | Нет закупочной цены              | `unit_cost = null`, строка не включается в полный итог |
| `STALE_PURCHASE_PRICE`    | Использована цена старше 14 дней | Расчёт выполняется, показывается дата цены             |
| `ZERO_PURCHASE_PRICE`     | Цена равна нулю                  | Требует явного признака, что нулевая цена допустима    |
| `MISSING_RECIPE`          | Нет рецептуры                    | Себестоимость блюда не рассчитывается                  |
| `MISSING_RECIPE_VERSION`  | На дату нет версии               | Не применять текущую версию молча                      |
| `RECIPE_CYCLE`            | Цикл полуфабрикатов              | Остановить строку и показать путь цикла                |
| `UNIT_CONVERSION_MISSING` | Нет коэффициента единицы         | Не выполнять предположительное преобразование          |
| `ESTIMATED_PRODUCTION`    | Нет документов выпуска           | Показать, что выпуск расчётный                         |
| `PARTIAL_COST`            | У одного из компонентов нет цены | Общая себестоимость блюда `null`                       |

В интерфейсе предупреждения объединяются под аккордеоном. В заголовке — число кафе, позиций и критических ошибок; внутри — группировка по кафе и позиции с раскрытием конкретных документов и формул.

### 11.6. Поведение таблицы

- Числа выравнивать вправо и показывать табличными цифрами.
- Названия выравнивать влево, разрешить перенос на две строки без обрезки важных слов.
- Первая колонка и заголовок фиксированы при прокрутке.
- Ширина числовых колонок задаётся по типу показателя; минимальная ширина должна вмещать заголовок, значение и иконку детализации.
- При большом числе колонок использовать горизонтальную прокрутку, а не сжимать данные до наложения.
- Для сортируемых заголовков использовать полноценную кликабельную область и `aria-sort`.
- На мобильном показывать карточки строк или сокращённый набор ключевых колонок с раскрытием остальных показателей.
- Отрицательные значения выделять красным, положительные — зелёным, но знак и текст должны сохраняться: цвет не является единственным носителем смысла.
- Денежный формат: `12 345,67 ₽`; количество — до 3 знаков после запятой; нулевые незначащие разряды можно скрывать.
- Расчёты вести без раннего округления; округлять только при отображении.

## 13. Проверка расчётов

### 13.1. Unit-тесты

- взвешенная цена двух поставок разного объёма;
- возврат поставщику и корректировка цены/количества;
- отсутствие поступлений в 14-дневном окне и fallback на последнюю цену;
- полное отсутствие цены;
- холодные и горячие потери;
- отрицательный процент, создающий привес;
- вложенная рецептура в несколько уровней;
- один продукт по нескольким веткам рецептуры;
- цикл рецептур;
- несовместимые единицы измерения;
- версия рецептуры, сменившаяся в середине периода;
- нулевые продажи;
- итоговая наценка из сумм, а не среднее процентов.

### 13.2. Контрольный пример из Excel

Для одной единицы ролла ожидается приблизительно:

```text
рис     0,070 кг
вода    0,074 кг
уксус   0,009 кг
сахар   0,009 кг
соль    0,001 кг
итог    3,39 ₽/шт
```

Допуск сравнения должен учитывать, что книга округляет промежуточные количества. Эталон backend нужно считать с полной точностью и округлять только конечное отображение.

### 13.3. Интеграционные сценарии

- одно и несколько кафе;
- товар для перепродажи и готовое блюдо;
- фактический и расчётный выпуск;
- блюдо с частично отсутствующими ценами;
- кафе с разными закупочными ценами;
- изменение рецептуры в периоде;
- права на просмотр и экспорт;
- XLS совпадает с экранным итогом.

## 14. Решения, которые нужно подтвердить до backend-разработки

1. Цена за 14 дней фиксируется один раз перед началом всего отчёта либо пересчитывается на дату каждого выпуска? Для корректной фактической себестоимости рекомендуется второй вариант; таблица показывает первый, упрощённый вариант.
2. Используется цена с НДС или без НДС?
3. Какие статусы `billing` считаются проведёнными и как кодируются возвраты/корректировки?
4. Где хранятся документы фактического выпуска продукции? Если их нет, можно ли официально использовать оценку `продажи + списания`?
5. Учитывается ли остаток готовой продукции на начало и конец периода?
6. Какие таблицы и поля хранят версии рецептур и проценты холодных/горячих потерь?
7. Нулевая цена воды — допустимая бизнес-настройка или отсутствие данных?
8. Как в справочнике однозначно отличить товар для перепродажи от конечного блюда?
9. Входит ли стоимость упаковки, труда, доставки и производственных накладных расходов? Сейчас материалы содержат только стоимость ингредиентов.
10. Нужно ли считать отчёт по дням выпуска с последующим суммированием или допустима одна цена на весь период?

## 15. Лучшие практики, использованные в спецификации

- IFRS IAS 2 допускает взвешенную среднюю стоимость для взаимозаменяемых запасов и включает в себестоимость затраты на приобретение и переработку: [IAS 2 Inventories](https://www.ifrs.org/issued-standards/list-of-standards/ias-2-inventories/).
- При среднем методе стоимость должна быть взвешена количеством; поступления и корректировки должны менять стоимость запасов: [Oracle Average Costing](https://docs.oracle.com/cd/A60725_05/html/comnls/us/cst/avgov.htm).
- В производственном расчёте себестоимость партии строится из количества и стоимости потреблённых ингредиентов, затем делится на выпуск партии: [Oracle Process Manufacturing Cost Management](https://docs.oracle.com/cd/E26401_01/doc.122/e49061/T388436T388443.htm).
- Выход и потери нужно учитывать на уровнях технологического процесса и рецептуры: [Oracle Process Manufacturing Yield](https://docs.oracle.com/en/cloud/saas/readiness/scm/25d/inv25d/25D-inventory-wn-f40557.htm).
- Для MUI-таблицы поддерживаются sticky header, группировка заголовков, сортировка и collapsible rows: [Material UI Table](https://mui.com/material-ui/react-table/).
- Сложная таблица должна сохранять семантическую связь заголовков и ячеек; многоуровневые заголовки требуют `scope`, `id`/`headers`: [W3C Tables Tutorial](https://www.w3.org/WAI/tutorials/tables/).
- Сортировка должна отражаться через `aria-sort`, а кнопка сортировки занимать всю область заголовка: [W3C Sortable Table](https://www.w3.org/WAI/ARIA/apg/patterns/table/examples/sortable-table/).

- кнопка закрытия, диалог прокручивается внутри и не превышает высоту viewport.

Диалог «Проверка расчёта» для блюда:
- название блюда, кафе, период;
- чип «Фактический выпуск» либо предупреждающий чип «Расчётный выпуск»;
- количество выпуска, продаж и списаний;
- версия и дата действия рецептуры;
- дерево рецептуры: блюдо → полуфабрикаты → исходные продукты;
- у каждого узла показать брутто, холодные потери %, нетто, горячие потери %, выход;
- для исходного продукта показать количество, закупочную цену, стоимость и источник цены;
- под деревом показать полную подстановку значений и итоговую себестоимость единицы;
- отсутствующие цены показывать как «—», не как 0.

- одно кафе с расчётным выпуском;
- нулевые, положительные и отрицательные показатели валовой прибыли.

Обязательные состояния:

- пустой результат;

Типы данных:

## 17. Вывод

Материалы достаточны для проектирования интерфейса и прототипа расчёта, но недостаточны для безопасного production-backend без подтверждения источника фактического выпуска, версий рецептур, статусов поступлений и трактовки 14-дневной цены. Эти решения влияют на сами цифры, поэтому их нужно зафиксировать до реализации расчётного сервиса.
